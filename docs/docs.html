rgsplat async

High-performance 3D Gaussian Splatting from video — Rust 2021 · GPU + CPU fallback · Tokio async · Rayon parallel

On this page
  • Installation
  • Quick start
  • Pipeline overview

This crate implements the complete pipeline described in 3D Gaussian Splatting for Real-Time Radiance Field Rendering (Kerbl et al., SIGGRAPH 2023). It takes a video as input and produces a trained .ply point cloud of 3D Gaussians that can be rendered in real time.

Design principles

Principle How
Never block the async executor All CPU-heavy work runs in tokio::task::spawn_blocking. Subprocesses (ffmpeg, COLMAP) use tokio::process::Command.
GPU-optional GPU support is a Cargo feature (--features gpu). Falls back to CPU automatically if no adapter found.
Memory efficiency Gaussian3D is #[repr(C)] and implements bytemuck::Pod — zero-copy GPU uploads. Large files use memmap2.
Correctness first Quaternions are renormalized every Adam step. Covariance is always positive semi-definite (RS·RSᵀ construction).
Parallel by default Rayon par_iter over Gaussians for projection, gradient update, densification. Tile rasterizer uses zero-contention tile decomposition.

Installation

Prerequisites and build configurations

Quick install (one command)

Installs Rust, ffmpeg, COLMAP, and builds rgsplat:

Linux / macOS

bash <(curl -fsSL https://raw.githubusercontent.com/mfaeezshabbir/rgsplat/main/scripts/install.sh)

Windows (PowerShell as Admin)

powershell -ExecutionPolicy Bypass -File https://raw.githubusercontent.com/mfaeezshabbir/rgsplat/main/scripts/install.ps1

Or clone and run locally:

git clone https://github.com/mfaeezshabbir/rgsplat.git
cd rgsplat
bash scripts/install.sh          # Linux / macOS
powershell -File scripts/install.ps1   # Windows

Quick start

From video to trained Gaussians in one call

CLI

    rgsplat --input my_scene.mp4 --output ./output
    
    # GPU mode, 2fps extraction, 30k iterations
    rgsplat --input my_scene.mp4 --output ./output --gpu --fps 2 --iters 30000
    
    # Override training hyper-parameters
    rgsplat --input scene.mp4 --output ./out \
      --gpu \
      --lr-position 0.00016 \
      --densify-from 500 \
      --densify-until 15000 \
      --max-gaussians 2000000

    rgsplat --help

Pipeline stages

What happens inside Pipeline::run()

Stage 1 — Frame extraction

Calls ffmpeg as an async subprocess. Non-blocking: the Tokio runtime handles other tasks while ffmpeg runs.

sfm::extract_frames(&video_path, &frames_dir, fps).await?
Parameter Default Notes
fps 2.0 2 frames/sec balances coverage vs COLMAP cost. Increase for fast-moving scenes.
Output format PNG (-qscale:v 1) Highest quality; lossless SIFT features.

Stage 2 — Structure from Motion (COLMAP)

Runs three COLMAP passes in sequence, all as async subprocesses:

Pass COLMAP command What it does
1 feature_extractor SIFT keypoint detection on every frame
2 exhaustive_matcher Match features between all frame pairs
3 mapper Incremental SfM → camera poses + sparse 3D points

COLMAP is the main bottleneck. For a 100-frame sequence, expect 5–15 minutes on CPU. Use --gpu to enable GPU SIFT and GPU matching — roughly 5× faster.

Stage 3 — Gaussian initialisation

Parses COLMAP binary files (points3D.bin, cameras.bin) and converts each SfM point into a Gaussian3D:

Parameter Initial value Derivation
Position SfM xyz Direct copy
Color (SH DC) SfM RGB (linear(r) - 0.5) × SH_C0_INV
Scale (isotropic) mean NN distance K=3 nearest neighbours via spatial sort
Rotation identity quaternion [1, 0, 0, 0]
Opacity logit(0.1) ≈ −2.2 10% opacity, optimised during training

Stage 4 — Training loop

30 000 iterations (default). Each iteration:

Pick random camera → Render (GPU or CPU) → Compute L1+D-SSIM loss
    → Backward pass (per-Gaussian gradients) → Adam step → [Densify/Prune every 100 iters]

Key events during training:

Iteration range Event
500 – 15 000 Adaptive densification window (clone + split)
Every 3 000 Opacity reset — forces re-learning of which Gaussians are real
Every 5 000 Checkpoint save to ckpt_NNNNNN.ply (async, non-blocking)

Stage 5 — Export

Saves a PLY point cloud where each Gaussian is a point with custom properties (position, rotation, scale, opacity, 48 SH coefficients). Compatible with standard Gaussian Splatting viewers.

Memory model

How data is stored and moved through the pipeline

Gaussian3D memory layout

Gaussian3D is #[repr(C)] with explicit field ordering to minimise padding and allow zero-copy GPU uploads via bytemuck::Pod.

Field Type Bytes Purpose
position [f32; 3] 12 World-space centre μ
opacity_logit f32 4 Pre-sigmoid opacity (avoids gradient vanishing)
rotation [f32; 4] 16 Unit quaternion [w,x,y,z]
log_scale [f32; 3] 12 Log scale — exp gives positive scale
(padding) — 4 Align to 16B for GPU vec4 alignment
sh_coeffs [f32; 48] 192 16 SH coefficients × 3 RGB channels
Total 240B

Allocation strategy

Data Strategy Why
Gaussian array Vec<Gaussian3D> Contiguous — good cache locality, SIMD-friendly, direct GPU upload with bytemuck::cast_slice
Adam state (m1/m2) Vec<f32> flat arrays Indexed as [gaussian_idx * 59 + param_idx] — avoids pointer indirection
Framebuffer Vec<f32> RGBA Single allocation, passed to GPU or tile rasterizer
Large binary files (COLMAP) memmap2::Mmap OS-managed paging — only touched pages loaded into RAM
Tile pixel buffers Stack ([f32; 16×16×4]) Per-rayon-thread, no heap allocation in hot loop

GPU buffer flow

Vec<Gaussian3D>
      → bytemuck::cast_slice (zero-copy view)
      → wgpu STORAGE buffer (COPY_DST | COPY_SRC)
      → compute shader: project pass
      → SplatInfo2D STORAGE buffer
      → radix sort (depth keys)
      → compute shader: rasterize pass
      → texture_storage_2d (rgba8unorm)
      → readback buffer (MAP_READ)
      → Vec<u8> on CPU

GPU buffer uploads use DeviceExt::create_buffer_init — one memcpy from CPU slice to VRAM, no intermediate staging buffer allocation needed in application code.

Concurrency model

How the pipeline stays non-blocking end-to-end

Thread roles

Thread pool Used for Size
Tokio async executor Orchestration, subprocess I/O, GPU submission futures Tokio default (num CPUs)
Tokio blocking pool CPU-heavy Rust code: projection, Adam step, densification, parsing 512 (Tokio default max)
Rayon thread pool Data parallelism within a blocking task: par_iter over Gaussians, tile rasterizer num CPUs

Key async boundaries

// This blocks the thread for ~minutes — must be spawn_blocking
    let (gaussians, cameras) = tokio::task::spawn_blocking(move || {
        load_sfm_output(&workspace)
    }).await??;
    
    // Render runs in blocking pool, GPU future awaits in async context
    let rendered = task::spawn_blocking(move || {
        cpu::rasterizer::rasterize(&gaussians, &camera, bg)
    }).await?;
    
    // Checkpoint saves don't block the training loop
    task::spawn(async move {
        task::spawn_blocking(move || save_ply(&g, &path)).await
    });

Tile rasterizer zero-contention design

The screen is divided into 16×16 tiles. Each tile writes to a disjoint range of the output pixel buffer. Rayon iterates over tiles in parallel with no mutexes needed — a single unsafe pointer write is used with a safety invariant that's trivially verified (each pixel belongs to exactly one tile).

The unsafe tile write is the only unsafe block in the codebase. It replaces what would otherwise be per-pixel Mutex contention with a guaranteed-disjoint memory partition.

Gaussian primitives

The mathematical object at the core of 3DGS

The 3D Gaussian function

Each scene primitive is a 3D Gaussian density function centred at μ with covariance Σ:

G(x) = exp(−½ (x − μ)ᵀ Σ⁻¹ (x − μ))

Covariance factorisation

Σ is parameterised as the product of a rotation matrix R and a diagonal scale matrix S, ensuring Σ is always positive semi-definite:

Σ = R · S · Sᵀ · Rᵀ = (RS)(RS)ᵀ

In code, R is stored as a unit quaternion and S as log-scale (so optimisation can take unconstrained steps):

// Decode scale: always positive
    s = [log_scale[0].exp(), log_scale[1].exp(), log_scale[2].exp()]
    
    // Decode opacity: always in (0, 1)
    α = sigmoid(opacity_logit) = 1 / (1 + exp(−opacity_logit))

Why logit/log parameterisation?

Adam takes unconstrained gradient steps. If we stored opacity directly as a float in (0,1), a gradient step could push it outside the valid range. Storing the logit (inverse sigmoid) means any real number is valid, and we decode to (0,1) at evaluation time. Same for scale — storing log-scale means any real number decodes to a positive scale via exp.

Trainable parameters per Gaussian

Parameter Raw storage Decoded form Count
Position μ f32[3] World xyz 3
Rotation f32[4] quaternion 3×3 rotation matrix R 4
Scale f32[3] log exp → positive scale 3
Opacity f32 logit sigmoid → (0,1) 1
SH coefficients f32[48] 16 coeffs × 3 channels 48
Total 59

Covariance projection (EWA)

Mapping 3D Gaussian ellipsoids to 2D screen splats

The EWA splatting formula

Based on Zwicker et al. 2001. The 2D screen covariance is:

Σ′₂D = J · W · Σ₃D · Wᵀ · Jᵀ

where W is the view rotation (upper-left 3×3 of the view matrix) and J is the affine approximation Jacobian of the perspective projection at the Gaussian centre:

J = [ fx/tz 0 −fx·tx/tz² ]
[ 0 fy/tz −fy·ty/tz² ]
[ 0 0 0 ]

Conic representation

For rasterization, we store the inverse 2×2 covariance (the conic) to avoid a matrix inverse at every pixel:

Σ⁻¹ = 1/det · [ c −b ] where Σ = [ a b ]
[ −b a ] [ b c ]

Gaussian evaluation at pixel (dx, dy) from the centre:

power = −½ (a·dx² + 2b·dx·dy + c·dy²)
alpha = σ(opacity_logit) · exp(power)

Low-pass anti-aliasing filter

A 0.3-pixel constant is added to the diagonal of Σ′₂D before inversion. This prevents Gaussian splats from becoming sub-pixel (infinitely sharp) and causing aliasing:

sigma_2d[(0,0)] += 0.3;
    sigma_2d[(1,1)] += 0.3;

Screen-space AABB

The tight bounding box of the Gaussian splat (n_sigma = 3, covering 99.7% of density) is computed from the eigenvalues of the 2×2 covariance:

mid = (a + c) / 2
    disc = sqrt(((a−c)/2)² + b²)
    λ₁ = mid + disc  (largest eigenvalue)
    λ₂ = mid − disc
    r = n_sigma × sqrt(max(λ₁, λ₂))

Spherical harmonics

View-dependent colour via degree-3 real SH basis

What SH gives us

Each Gaussian can emit different colour depending on the viewing direction — mimicking specular highlights, subsurface scattering, and non-Lambertian surfaces. The colour for view direction d is:

c(d) = Σ_{l=0}^{3} Σ_{m=−l}^{l} k_{lm} · Y_{lm}(d)

Coefficient layout

48 floats per Gaussian: 16 SH coefficients × 3 RGB channels, stored channel-major:

sh_coeffs[0..16]  = Red channel (l=0 to l=3)
    sh_coeffs[16..32] = Green channel
    sh_coeffs[32..48] = Blue channel

Degree 0 (DC term)

The degree-0 term gives a view-independent base colour:

c_dc = 0.5 + SH_C0 · k₀₀, SH_C0 = 1 / (2√π) ≈ 0.2821

Constants

Degree Count Constant(s)
l=0 1 SH_C0 = 0.28209479
l=1 3 SH_C1 = 0.48860251
l=2 5 SH_C2[0..5]
l=3 7 SH_C3[0..7]

Learning rate

Higher-degree SH coefficients use a 20× smaller learning rate than the DC term, preventing high-frequency colour noise from appearing early in training:

lr_sh_dc   = 0.0025
    lr_sh_rest = lr_sh_dc / 20.0 = 0.000125

Camera & projection

Pinhole camera model, NDC mapping, frustum culling

Pinhole camera model

Symbol Field Meaning
fx, fy Camera.fx / fy Focal lengths in pixels
cx, cy Camera.cx / cy Principal point (image centre)
W, H Camera.width / height Image resolution in pixels
R, t Camera.c2w upper 3×3, col 3 Camera-to-world rotation + translation
k1, k2, k3 Camera.distortion Radial distortion (currently unused in rasterizer)

Perspective projection

World point P → screen pixel (sx, sy):

p_cam = view · [P, 1]ᵀ
sx = fx · p_cam.x / p_cam.z + cx
sy = fy · p_cam.y / p_cam.z + cy

NDC projection matrix

OpenCV-convention pinhole camera to OpenGL NDC (for wgpu):

[ 2fx/W 0 (W−2cx)/W 0 ]
[ 0 −2fy/H (H−2cy)/H 0 ]
[ 0 0 −(f+n)/(f−n) −2fn/(f−n) ]
[ 0 0 −1 0 ]

Frustum culling

Six planes extracted from the MVP matrix (Gribb-Hartmann method). Sphere test using Gaussian position and bounding radius keeps only visible Gaussians. Reduces GPU/CPU work proportionally to scene complexity.

Depth key encoding

Camera-space depth (positive float) is encoded as a sortable u32 by bit-flipping. This allows a standard ascending radix sort to produce back-to-front order:

// In WGSL
    depth_keys[idx] = ~bitcast<u32>(tz);  // flip all bits → back-to-front order

TrainConfig

All hyper-parameters with defaults from the 3DGS paper

num_iterations
Default: 30_000
Total training steps. 15k gives decent results; 30k is the paper default.
lr_position
Default: 0.00016
Adam learning rate for Gaussian centres. Very small — position is sensitive.
lr_rotation
Default: 0.001
Learning rate for quaternion components.
lr_scale
Default: 0.005
Learning rate for log-scale axes.
lr_opacity
Default: 0.05
Largest LR — opacity converges fast.
lr_sh_dc
Default: 0.0025
Learning rate for degree-0 SH (view-independent colour).
lr_sh_rest
Default: 0.000125
20× smaller than lr_sh_dc. Prevents early high-frequency colour noise.
beta1
Default: 0.9
Adam first moment decay. Standard value.
beta2
Default: 0.999
Adam second moment decay.
eps
Default: 1e-8
Adam denominator stability term.
densify_from_iter
Default: 500
Start densification after this many iterations (let positions settle first).
densify_until_iter
Default: 15_000
Stop densification — Gaussians should be well-placed by now.
densify_grad_threshold
Default: 0.0002
Average 2D screen gradient norm threshold to trigger clone/split.
opacity_reset_iter
Default: 3000
Reset all opacities to 0.01 every N iters — forces re-learning, kills floaters.
prune_opacity_thresh
Default: 0.005
Remove Gaussians with opacity below this after each densification.
max_gaussians
Default: 3_000_000
Hard cap on total Gaussians. Densification stops when this is reached.

GPU vs CPU guide

Choosing the right compute backend for your environment

Decision matrix

Scenario Recommended
Training speed (interactive feedback) GPU always
Headless CI / render farm node CPU (no GPU driver requirement)
macOS Apple Silicon GPU (wgpu Metal backend, very fast)
Low-memory device (<4GB VRAM) CPU (avoids VRAM OOM on large scenes)
Production server (no display) CPU with Rayon (predictable)

Automatic fallback

let pipeline = Pipeline::new(cfg).await?;
    // If GPU init fails, Pipeline logs a warning and continues with CPU.
    // No code change needed.

Manual override

let available = gpu::gpu_available().await;
    cfg.use_gpu = available && std::env::var("GSPLAT_NO_GPU").is_err();

Python bindings

Using the pipeline from Python via PyO3 / maturin

Build the wheel

pip install maturin
    maturin build --release --features python
    pip install target/wheels/rgsplat-*.whl

Usage

import rgsplat as gs
    import asyncio
    
    async def main():
        cfg = gs.PipelineConfig("scene.mp4", "./output")
        cfg.use_gpu = True
        cfg.train.num_iterations = 15000
    
        pipeline = await gs.Pipeline.new(cfg)
        output_path = await pipeline.run()
        print(f"Done: {output_path}")
    
    asyncio.run(main())

The Python bindings expose an async interface via PyO3's pyo3-asyncio integration. The Tokio runtime is created once and shared across all pipeline calls in the process.

Memory usage estimate

// Rough formula:
    // RAM = N × (240B Gaussian + 2 × 59 × 4B Adam state)
    // For 1M Gaussians: 1M × (240 + 472) ≈ 680MB RAM
    // GPU VRAM: same for buffers + framebuffer (W×H×16B)

Adam state is the largest allocation: N × 59 × 2 × 4 bytes (m1 + m2). For 3M Gaussians this is ~1.4GB. If RAM is tight, reduce max_gaussians or switch to a half-precision Adam variant.
